## 2. Automotive In-Vehicle Network 변화

차량에는 ADAS, Charging Vehicle Platform, Connected Car 같은 기능이 계속 추가되고 있다. 이 기능들은 카메라, 레이더, 라이다, 인포테인먼트, 배터리/전력 제어처럼 많은 데이터를 생산하고, 기존 차량 내부 네트워크보다 높은 대역폭과 낮은 지연을 요구한다.

기존 구조에서는 기능별 ECU가 분산되어 있고, Gateway가 여러 bus를 연결한다. 이 구조는 기능이 늘어날수록 배선과 통신 경로가 복잡해진다. 최근에는 Domain-based architecture와 Zonal-based architecture로 이동하면서, 차량을 물리 구역 또는 기능 구역별로 묶고 고속 backbone을 통해 데이터를 모으는 방향으로 발전한다.

### 기존 네트워크와 Ethernet 비교

| 항목 | 기존 CAN 중심 네트워크 | Ethernet 기반 네트워크 |
| --- | --- | --- |
| 전송 속도 | CAN 기준 약 500 kbps 수준 | 10 Mbps부터 10 Gbps까지 확장 |
| payload | CAN classic 기준 8 bytes | Ethernet payload 최대 1500 bytes |
| 장점 | 제어 신호에 적합, 구조 단순 | 고대역폭 센서/멀티미디어/백본에 적합 |
| 한계 | 대용량 데이터와 복잡한 서비스 처리 어려움 | 기본 Ethernet만으로는 실시간성 보장 어려움 |

Ethernet은 고속 전송에 유리하지만, 일반 Ethernet은 CSMA/CD 기반 특성상 frame 충돌과 지연 변동이 발생할 수 있다. 차량 제어 데이터에는 정해진 시간 안에 도착해야 하는 메시지가 많기 때문에, Ethernet을 차량 제어망에 쓰려면 실시간성을 보장하는 TSN이 필요하다.

## 3. TSN이 필요한 이유

차량 네트워크에는 성격이 다른 traffic이 섞여 있다.

| Traffic 종류 | 예시 | 요구사항 |
| --- | --- | --- |
| 제어 데이터 | brake, steering, chassis, safety control | 낮은 지연, 지연 상한 보장 |
| 센서 데이터 | camera, radar, lidar | 높은 대역폭, 안정적인 전송 |
| 멀티미디어 | audio, video, infotainment | 일정한 품질과 대역폭 |
| 일반 데이터 | logging, diagnostic, update | 상대적으로 낮은 우선순위 |

TSN(Time-Sensitive Networking)은 하나의 기술이 아니라 IEEE 802.1 계열의 여러 표준을 묶은 개념이다. Ethernet 기반 네트워크에서 시간 동기화, 지연 보장, traffic scheduling, frame redundancy, path control 등을 제공해 차량 제어망에서도 Ethernet을 사용할 수 있게 한다.

대표적인 TSN 관련 표준:

| 표준 | 내용 | 핵심 역할 |
| --- | --- | --- |
| IEEE 802.1AS | Timing and Synchronization | 장치 간 시간 동기화 |
| IEEE 802.1Qav | Credit Based Shaper | AVB 계열 traffic shaping |
| IEEE 802.1Qbv | Time-Aware Shaper | 시간 슬롯 기반 gate scheduling |
| IEEE 802.1Qbu / IEEE 802.3br | Frame Preemption | 긴 frame 때문에 긴급 frame이 밀리는 문제 완화 |
| IEEE 802.1Qch | Cyclic Queuing and Forwarding | cycle 기반 deterministic forwarding |
| IEEE 802.1CB | Frame Replication and Elimination for Reliability | 복제/중복 제거 기반 신뢰성 향상 |
| IEEE 802.1Qci | Per-Stream Filtering and Policing | stream 단위 filtering/policing |
| IEEE 802.1Qca | Path Control and Reservation | 경로 제어와 자원 예약 |
| IEEE 802.1Qcr | Asynchronous Traffic Shaping | 동기 시간 슬롯 없이 shaping |

## 4. Ethernet과 OSI 계층 복습

OSI 7계층은 통신 기능을 논리적으로 나눈 모델이다.

| 계층 | 역할 | 예시 |
| --- | --- | --- |
| Application | 응용 서비스 | HTTP, FTP, SOME/IP application |
| Presentation | 표현/인코딩 | JPEG, MPEG, TLS |
| Session | 세션 관리 | socket, API |
| Transport | end-to-end 전송 | TCP, UDP |
| Network | logical addressing/routing | IP |
| Data-Link | frame, MAC addressing | Ethernet MAC, VLAN |
| Physical | 신호와 매체 | Ethernet PHY |

송신 측에서는 상위 계층 데이터에 각 계층 header가 붙는 encapsulation이 일어나고, 수신 측에서는 header를 하나씩 해석하는 decapsulation이 일어난다.

계층별 packet 이름:

| 계층 | packet 이름 |
| --- | --- |
| Application | Message |
| Transport TCP | TCP Segment |
| Transport UDP | UDP User Datagram |
| Network | Datagram |
| Data-Link | Frame |
| Physical | Bit |

## 5. Ethernet frame 필드

Ethernet frame은 다음 필드로 구성된다.

| 필드 | 크기 | 의미 |
| --- | --- | --- |
| Preamble | 7 bytes | 수신 동기화 |
| SFD | 1 byte | Start Frame Delimiter |
| Destination MAC | 6 bytes | 목적지 MAC 주소 |
| Source MAC | 6 bytes | 출발지 MAC 주소 |
| Type | 2 bytes | 상위 protocol 식별 |
| Data | 46-1500 bytes | payload |
| CRC/FCS | 4 bytes | 오류 검출 |

대표 EtherType:

| 값 | 의미 |
| --- | --- |
| `0x0800` | IPv4 |
| `0x88F7` | PTP over Ethernet |
| `0x8100` | VLAN tagged frame |

## 6. VLAN frame 구조

VLAN은 Ethernet frame에 802.1Q tag를 추가해 논리 네트워크와 우선순위를 표현한다.

| 필드 | 크기 | 의미 |
| --- | --- | --- |
| TPID | 16 bits | VLAN tag 식별자, 보통 `0x8100` |
| PCP | 3 bits | Priority Code Point |
| DEI/CFI | 1 bit | drop eligible 또는 format indicator |
| VID | 12 bits | VLAN ID |

TSN에서는 VLAN PCP 값이 traffic priority 분류에 자주 사용된다. 예를 들어 Scheduled Traffic, Multimedia Traffic, Best Effort Traffic을 서로 다른 PCP로 분류하고 switch 내부 queue에 배치할 수 있다.

## 7. IP, UDP, PTP packet

PTP는 Ethernet 위에 직접 실릴 수도 있고, IPv4/UDP 위에 실릴 수도 있다.

PTP over Ethernet:

```text
Ethernet Header -> PTP Header -> PTP Payload
```

PTP over UDP/IPv4:

```text
Ethernet Header -> IP Header -> UDP Header -> PTP Header -> PTP Payload
```

Wireshark에서는 frame list, packet detail, packet bytes 영역을 함께 확인하며 PTP header field와 timestamp 위치를 분석한다.

## 8. AVB와 시간 동기화

AVB(Audio Video Bridging)는 Ethernet 기반으로 오디오/비디오 traffic을 안정적으로 전달하기 위한 표준 묶음이다. TSN은 AVB에서 확장되어 더 넓은 산업/차량 제어 traffic까지 다루는 방향으로 발전했다.

시간 동기화가 중요한 이유는 여러 ECU와 switch가 동일한 시간 기준을 공유해야 scheduling이 가능하기 때문이다. TAS처럼 gate를 특정 시간에 열고 닫는 기법은 장치마다 시간이 다르면 오히려 지연이나 충돌을 만들 수 있다.

## 9. PTP와 IEEE 802.1AS

PTP(Precision Time Protocol)는 네트워크 장치 간 시간을 정밀하게 맞추기 위한 protocol이다. IEEE 1588 기반이며, 차량 Ethernet/TSN에서는 IEEE 802.1AS가 시간 동기화 표준으로 사용된다.

PTP에서는 Master clock과 Slave clock이 존재한다. Slave는 Master가 보낸 시간 정보를 바탕으로 자신의 시간을 보정한다.

대표적인 PTP 메시지:

| 메시지 | 역할 |
| --- | --- |
| Sync | Master가 동기화 기준 시간을 전달 |
| Follow_Up | Sync의 정확한 timestamp를 후속 전달 |
| Delay_Req | Slave가 Master로 지연 측정을 요청 |
| Delay_Resp | Master가 Delay_Req 수신 시간을 응답 |

PTP 동기화에서는 송수신 시각의 timestamp를 이용해 offset과 path delay를 계산한다. 이를 통해 Slave clock을 Master clock에 맞춘다.

## 10. PTP packet 분석 포인트

실습에서는 Wireshark로 PTP packet을 캡처하고 다음을 확인한다.

- PTP packet이 Ethernet type `0x88F7` 또는 UDP/IP encapsulation으로 들어오는지 확인
- PTP header의 message type과 sequence ID 확인
- Sync, Follow_Up, Delay_Req, Delay_Resp가 어떤 순서로 전달되는지 확인
- packet bytes 영역에서 timestamp가 어떤 byte 구조로 들어가는지 확인
- 시간 동기화 전후 Slave 시간이 Master와 어떻게 맞춰지는지 확인

## 11. CQF - Cyclic Queuing and Forwarding

CQF(IEEE 802.1Qch)는 공통 시간 cycle을 기준으로 frame을 queue에 넣고 다음 cycle에 forwarding하는 방식이다. 일반적으로 두 개 이상의 buffer를 사용해 한 cycle에서는 수신, 다음 cycle에서는 송신을 수행한다.

핵심 개념:

- network 전체가 common time cycle을 공유한다.
- frame은 cycle 단위로 queueing되고 forwarding된다.
- cycle boundary를 기준으로 처리하므로 jitter를 낮출 수 있다.
- deterministic latency를 만들기 위한 TSN 기법 중 하나다.

## 12. IEEE 802.1CB - 신뢰성 향상

IEEE 802.1CB는 FRER(Frame Replication and Elimination for Reliability)을 제공한다. 중요한 stream을 여러 경로로 복제해 보내고, 수신 측에서 중복 frame을 제거한다. 하나의 경로에 문제가 생겨도 다른 경로로 frame이 도착하면 통신을 유지할 수 있다.

### Stream Identification

stream을 식별하기 위해 다음 정보가 사용될 수 있다.

- Null stream identification
- Source MAC + VLAN 기반 식별
- Active Destination MAC + VLAN 기반 식별
- IP stream identification

식별된 stream은 `stream_handle`로 관리된다.

### FRER 동작

| 단계 | 내용 |
| --- | --- |
| Sequence generation | frame에 sequence number 부여 |
| Replication | 동일 stream frame을 여러 경로로 복제 |
| Elimination | 수신 측에서 중복 frame 제거 |
| Recovery | 누락/중복/순서 문제를 관리하며 stream 복구 |

이 방식은 차량 제어처럼 신뢰성이 중요한 traffic에서 경로 장애에 대한 회복력을 높인다.

## 13. TAS - Time-Aware Shaper

TAS(Time-Aware Shaper, IEEE 802.1Qbv)는 TSN의 대표적인 traffic control 기법이다. message 종류별로 전송 가능한 시간을 나누고, switch queue의 gate를 시간표에 따라 열고 닫아 deterministic latency를 만든다.

TAS의 핵심은 전송 시간을 traffic class별로 분할하는 것이다.

예시:

```text
[Scheduled Traffic] [Multimedia Traffic] [Best Effort] [Guard Band] ...
```

### TAS 구성 요소

| 요소 | 의미 |
| --- | --- |
| Queue | priority별 frame 대기열 |
| Gate | queue에서 frame이 나갈 수 있는지 제어 |
| GCL | Gate Control List, gate open/close 시간표 |
| Cycle time | GCL이 반복되는 주기 |
| Guard band | 긴 frame이 다음 scheduled slot을 침범하지 않도록 둔 보호 구간 |

### Priority queue 예시

자료에서는 PCP 기반 priority queue가 예시로 나온다.

| Traffic class | Priority |
| --- | --- |
| NC(Network Control) | 7 |
| IC(Internetwork Control) | 6 |
| VO(Voice) | 5 |
| VI(Video) | 3 |
| BE(Best Effort) | 0 |

Incoming frame은 priority에 따라 queue에 들어가고, 각 queue의 gate는 GCL에 따라 open/close된다. gate가 열린 queue만 전송 가능하다.

### TAS에서 시간 동기화가 중요한 이유

TAS는 switch와 endpoint가 같은 시간 기준을 공유해야 정상 동작한다. 장치별 시간이 어긋나면 한쪽 switch에서는 scheduled traffic slot이 열려 있는데 다른 쪽에서는 닫혀 있을 수 있고, frame이 다음 cycle까지 대기하면서 지연이 생긴다.

시간 동기화와 GCL coordination이 맞으면, scheduled traffic은 지정된 시간에 맞춰 통과하고 지연이 줄어든다.

## 14. ATS - Asynchronous Traffic Shaping

ATS(Asynchronous Traffic Shaping, IEEE 802.1Qcr)는 TAS처럼 전역 시간 슬롯에 강하게 의존하지 않고 traffic을 shaping하는 방식이다. 자료에서는 Token Bucket Shaper와 eligibility time 개념이 함께 나온다.

핵심 포인트:

- 전역 schedule 기반이 아닌 비동기 shaping 방식
- 각 stream 또는 traffic class의 eligibility time을 계산
- burst traffic을 제어하고 queue delay를 관리
- TAS보다 시간 동기화 의존도가 낮지만, deterministic 제어 방식은 다르다.

## 17. Time Synchronization 실습 구성

Time Synchronization 실습은 4인 1조 기준으로 진행된다.

역할:

| 역할 | 설명 |
| --- | --- |
| Master | 기준 clock 제공 |
| Slave 1 | Master에 동기화 |
| Slave 2 | Master에 동기화 |
| Slave 3 | Master에 동기화 |

각 보드는 Ethernet으로 연결하고, serial terminal을 통해 mode를 설정한다.

Mode 설정:

| 입력 | 역할 |
| --- | --- |
| `0` | Master |
| `1` | Slave 1 |
| `2` | Slave 2 |
| `3` | Slave 3 |

Slave mode에서는 Link Up과 Sync complete 상태를 LED 또는 terminal 출력으로 확인한다.

## 18. Time Synchronization Practice 1

목표: Master/Slave time synchronization 설정과 동기화 상태 확인

절차:

1. 모든 보드에 `RA6M3_PTP_Module.hex`를 다운로드한다.
2. 각 보드를 Ethernet으로 연결한다.
3. Tera Term에서 각 보드의 COM port를 연다.
4. 보드별 role을 입력한다.
5. Master mode와 Slave mode가 terminal에 정상 표시되는지 확인한다.
6. Slave 보드에서 Link Up과 Sync complete 상태를 확인한다.

확인 포인트:

- Master가 존재해야 Slave 시간이 기준 시간에 맞춰진다.
- Slave mode에서 동기화 완료 상태가 표시된다.
- 보드별 mode가 잘못 설정되면 동기화 흐름이 깨진다.

## 19. Time Synchronization Practice 2

목표: Master가 없을 때와 있을 때 Slave 동작 차이 확인

절차:

1. Slave 1, Slave 2, Slave 3를 먼저 Slave mode로 설정한다.
2. Master가 없는 상태에서 각 Slave time이 서로 다르게 흐르는지 확인한다.
3. 한 보드를 Master mode로 설정한다.
4. 일정 시간이 지난 뒤 Slave들의 시간이 Master 기준으로 맞춰지는지 확인한다.

해석:

- Master가 없으면 각 Slave는 자기 local clock 기준으로 동작한다.
- Master가 등장하면 PTP 메시지 교환을 통해 Slave clock이 보정된다.
- 실습의 핵심은 "시간 동기화 전후의 차이"를 눈으로 확인하는 것이다.

## 20. Time Synchronization Practice 3 - PTP Packet 분석

목표: Wireshark로 PTP packet을 캡처하고 header와 timestamp를 분석한다.

분석 순서:

1. Wireshark에서 Ethernet interface를 선택한다.
2. PTP 관련 packet을 capture한다.
3. packet list에서 PTP message 흐름을 확인한다.
4. packet detail에서 PTP header field를 확인한다.
5. packet bytes 영역에서 timestamp byte 구조를 확인한다.

질문형 확인 포인트:

- 시간 동기화 시 어떤 PTP 메시지가 전달되는가?
- timestamp는 packet byte 영역에서 어떤 구조로 들어가는가?
- message ID 또는 message type 값은 어떻게 구분되는가?
- Sync와 Follow_Up, Delay_Req와 Delay_Resp는 어떤 관계인가?

## 21. Time-Aware Shaper 실습 구성

TAS 실습은 Talker, Listener, switch board를 이용해 traffic shaping 적용 여부를 비교한다.

구성 예시:

| 장치 | 역할 | MAC address |
| --- | --- | --- |
| Talker 1 | traffic 송신 | `02:11:22:33:44:00` |
| Talker 2 | traffic 송신 | `02:11:22:33:44:01` |
| Listener 1 | traffic 수신 | `02:11:22:33:44:02` |
| Listener 2 | traffic 수신 | `02:11:22:33:44:03` |

Traffic 종류:

| Traffic | PCP | 의미 |
| --- | --- | --- |
| ST | 3 | Scheduled Traffic |
| MM | 1 | Multimedia Traffic |
| BE | 0 | Best Effort Traffic |

Talker는 ST/MM/BE traffic을 주기적으로 생성하고, Listener는 수신한 traffic count를 표시한다. RZN switch board는 TAS 설정과 PTP master/slave 설정을 담당한다.

## 22. RZN Switch와 TAS 설정

RZN switch board 설정:

| Switch | 기능 |
| --- | --- |
| ED0 | TAS Operation |
| ED2 | PTP On |
| ED3 | Master/Slave 설정 |

Serial terminal에서는 ST period를 입력한다.

예시:

- 기본 ST period: 40%
- `r`: ST period 재입력
- 80% 입력: scheduled traffic slot 비율 변경
- `e`: TAS 종료

TAS는 switch의 GCL과 PTP 동기화가 함께 맞아야 결과가 안정적으로 나온다.

## 23. TAS Practice 1 - TAS 적용

목표: PTP를 켜고 TAS를 적용한 상태에서 ST/MM/BE traffic 수신 패턴을 확인한다.

절차:

1. RZN SW1과 SW2의 ST period를 40%로 설정한다.
2. SW1을 Master, SW2를 Slave로 설정한다.
3. 두 switch 모두 PTP를 ON 상태로 둔다.
4. RA6 보드에서 Talker 1, Talker 2, Listener 1, Listener 2 mode를 설정한다.
5. Talker의 traffic period를 입력한다.
   - ST: 1
   - MM: 3
   - BE: 5
6. Listener에서 수신 count를 확인한다.
7. Talker를 시작하고 일정 시간 동안 결과를 기록한다.

확인 포인트:

- TAS가 적용되면 Scheduled Traffic이 지정된 시간 구간에서 우선적으로 처리된다.
- ST/MM/BE traffic의 수신 패턴이 GCL과 priority에 따라 달라진다.
- PTP가 켜져 있어야 switch 간 schedule 정렬이 가능하다.

## 24. TAS Practice 2 - TAS 미적용

목표: TAS를 끈 상태에서 동일 traffic 조건을 실행하고 결과를 비교한다.

절차:

1. Talker와 Listener를 reset한다.
2. RZN serial terminal에서 `e`를 입력해 TAS를 종료한다.
3. RA6 Talker/Listener를 다시 설정한다.
4. 동일한 ST/MM/BE traffic 조건으로 전송한다.
5. Listener 수신 count를 기록한다.

해석:

- TAS가 꺼지면 traffic이 시간 슬롯으로 보호되지 않는다.
- Scheduled Traffic도 다른 traffic과 queue 경쟁을 하게 된다.
- 지연과 jitter가 TAS 적용 상태보다 커질 수 있다.

## 26. 핵심 요약

- 차량 내부 네트워크는 ADAS, EV, Connected Car, SDV 방향으로 발전하면서 더 높은 대역폭과 deterministic latency를 요구한다.
- Ethernet은 고속 backbone에 적합하지만, 기본 Ethernet만으로는 차량 제어 traffic의 시간 보장을 설명하기 어렵다.
- TSN은 Ethernet 위에서 시간 동기화, traffic scheduling, frame redundancy, queue shaping 등을 제공하는 IEEE 표준 묶음이다.
- PTP/IEEE 802.1AS는 Master/Slave clock을 맞춰 TAS와 같은 시간 기반 제어가 가능하게 한다.
- IEEE 802.1CB는 frame 복제와 중복 제거로 신뢰성을 높인다.
- TAS(IEEE 802.1Qbv)는 Gate Control List로 queue gate를 시간별로 열고 닫아 Scheduled Traffic을 보호한다.
- TAS 실습에서는 PTP ON, Master/Slave switch 설정, ST/MM/BE traffic 생성, TAS 적용/미적용 결과 비교를 수행했다.

## 27. 면접에서 말하기 좋은 포인트

- 차량용 Ethernet을 단순 고속 통신이 아니라 TSN을 통해 실시간 제어 traffic까지 수용하는 backbone 기술로 설명할 수 있다.
- PTP 시간 동기화가 필요한 이유를 TAS의 gate schedule과 연결해 설명할 수 있다.
- Wireshark로 PTP packet을 직접 캡처하고, Sync/Follow_Up/Delay_Req/Delay_Resp 흐름과 timestamp 구조를 확인했다.
- TAS 실습에서 ST/MM/BE traffic을 나누고, PCP와 GCL 기반으로 scheduled traffic을 보호하는 과정을 확인했다.
- TAS 적용/미적용 결과를 graph로 비교해, deterministic networking이 왜 차량 제어망에서 중요한지 실습 기반으로 말할 수 있다.
